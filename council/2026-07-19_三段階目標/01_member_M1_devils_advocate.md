## 担当レンズ深掘り — 「最も甘い前提」10連発(全て file:line 付き)

### ① 「データが取れんときは安全側に倒れる」→ **完全に逆。欠損は全項目GO寄りにデフォルトされる**
`app.js:52-55` と `daily-brief.js:320-326` のフォールバック値:
- 波高欠損 → `?? 1.0`(m)= **8点**
- 風速欠損 → `?? 10`(km/h)= 2.78m/s = **10点**
- 天気欠損 → `?? 0`(快晴)= **10点**
- うねり欠損 → `?? 8`(秒)= **7点**

**全項目のデフォルトが「良好」寄り。** 安全設計の鉄則(fail-safe = 失敗したら危険側に倒す)の真逆になっとる。しかも `fetchAll` は `Promise.allSettled`(`api.js:155`)で個別失敗を握り潰すので、`main()` の `catch`(`app.js:123-127`)は**発火しない**。→ 詳細は「最も危険な一点」へ。

### ② 「気象庁の警報が出たら分かる」→ **スコアは警報を一切見ていない**
`calcScore`(`score.js:32`)の引数は `{ waveHeight, windSpeed, weatherCode, swellPeriod }` のみ。**警報・注意報は入力に存在しない。** 気象庁が**暴風警報**を出していても、Open-Meteoのモデルが凪を返せば、巨大リングは「8」、バナーは「✅ 出港OK」を表示し、その真横に「🌀暴風警報」チップが並ぶ。**アプリで最も権威ある危険信号(気象庁の警報)が、最も目立つ数字とバナーに一切反映されない。** 判断材料として本末転倒。

### ③ 「警報が無い＝安全」→ **取得失敗と平穏が区別不能**
`ui.js:174-178` `renderWarningChips`:items が空でも取得失敗でも `box.innerHTML = ''`。「警報チップが無い」画面は、(a)本当に平穏 (b)気象庁取得失敗 (c)stale判定 のどれか判別不能。しかも**Web版はstale状態すら出さない**——`daily-brief.js:82-87` のメールは「⚠️ 気象庁の警報データが更新停止中のため表示できません」と正直に出すのに、**Web の `renderWarningChips` には同等の分岐が無い**(`parsed.stale` を参照していない)。メールより本番Webの方が不誠実。

### ④ 「風は35%の重みで効く」→ **波が凪なら暴風でもGOになる(ハードキャップの非対称)**
`score.js:49-50` は波高スコア0(=波2.5m超)に**最大2点のハードキャップ**を掛ける。良い安全弁や。**が、風には同等のゲートが無い。** 手計算:波0.5m(=10点)・晴れ・うねり良好で、風20m/s(72km/h=気象庁「暴風」相当)でも → raw = 10×0.4+3×0.35+10×0.15+10×0.1 = 7.55、excess=0 → **8点「✅ 出港OK」**。波さえ凪なら暴風がマスクされる。船を出せるか・水面が荒れるかを実質決めるのは風なのに、波にだけ安全弁があって風に無いのは設計の穴。

### ⑤ 「スコアの数値基準は妥当」→ **完全な自作。実測にも事故データにも照合されていない**
`config.js:49-72` のしきい値(0.5m=10点、1.5m=6点、風15m/s=6点…)は開発者(本業FP&A・非専門家の自己申告)が置いた値。**海事・ダイビング事故統計・現場ガイドの合意との突き合わせ記録がリポジトリに一切無い。** 重み(波40/風35/天気15/うねり10)も同様。「バイブコーディング」を本人が認めているのは誠実だが、**その自作数字が「出港OK/困難」という命に関わる断定に化けている**のが問題。優くんの体感フィードバックは1人・主観・未記録。

### ⑥ 「ポイント別スコアは各ポイントの実態」→ **8点中5点は座標が推測、モデルはリーフ際に最も弱い**
`config.js:21-22` の TODO が正直に告白:「座標はおおよその位置」。ブリーフ確認済みは**トライアングルのみ**。黒島北・シュガーヒル・クエフ北は開発者の推定。加えて `ocean_current_velocity`(`api.js:120`)は約5km格子の全球モデル値で、**ダイバーが本当に知りたいリーフ際・地形性の潮流とは別物**。それを「潮流 0.3m/s(北)」と小数第1位まで、色付きチップ「8」で見せると、**モデルの限界を超えた精度を演出**してしまう。ドリフトダイブのリスク判断を誤らせ得る。

### ⑦ 「優くんは毎日見とる／そもそも使われとる」→ **計測ゼロ。前提が検証されていない**
リポジトリ全体にテレメトリ・アクセスログ・開封計測が**一切無い**。Webの閲覧回数もメール開封も未測定。「安全に貢献している」という主張の土台(=実際に見られている)が、希望的観測のまま。さらに唯一のプッシュである5時メールは **cron-job.org 単一依存**(`morning-brief.yml`:workflow_dispatchのみ)。cron-job.org か GitHub Actions が落ちれば**メールは来ず、誰も気づかない**(デッドマンズスイッチ=「今日は送れませんでした」通知が無い)。「今日は静かやな」と優くんが思った日が、実は配信障害の日かもしれん。

### ⑧ 「最終更新が新しい」→ **描画時刻を出しているだけ**
`ui.js:474-477` `renderFooter` は `new Date()`(=描画した瞬間の壁時計)を「最終更新」と表示。**元データが何時間前でも、キャッシュ由来でも、常に「たった今」に見える。** `renderDataInfo`(`ui.js:482-490`)は `weather.current.time` を使っていて誠実だが、フッターの方が目立つ位置で嘘をつく。

### ⑨ 「"今日の出港判断"スコア」→ **実際は1時間のスナップショット(メールは5時)**
`daily-brief.js:318-327` はcron発火時刻(5時JST)の `hIdx` で波・風を取る。ダイビングの出港・エントリーは8〜15時。**朝5時の海況を「今日の出港判断」と断定**している。Web版も現在時刻の1時間値(`app.js:51-56`)。時刻別テーブル(7-16時)は補助的にあるが、巨大な数字とバナーは1点の値。

### ⑩ 「免責がある／参考情報と位置づけている」→ **どこにも無い。むしろ責任を負う方向のコピーだけがある**
`grep -rn "免責|自己責任|参考|最終.*判断|責任|保証"` の結果、ヒットは `index.html:183` の「（お客様連絡の**参考**に）」1件のみ——免責ではなく週間カレンダーの用途説明。**「これは参考情報です」「最終判断はガイド自身が」「安全を保証するものではありません」の類が皆無。** 逆に `index.html:335` は「このアプリは優くんの**安全な**ダイビングガイド業務を**サポートするために作られました**」と、**安全ツールを自称**。責任を引き受ける方向のコピーだけがあって、免れる一文が無い。段階3で有料化した瞬間、この非対称が最悪化する(Q2で詳述)。

## 6問への回答

### Q1(重点)段階1の達成度／誤GOの残存経路
**達成度は「情報表示アプリ」としては高いが「安全判断の材料」としては未達。** 誤GO(危険なのにGOに見える)の経路が具体的に複数残っている:

1. **全API/一部API障害 → スコア8・出港OK**(前提①、最重要。次章)。cardは"--"でもリングとバナーはGOを叫ぶ。
2. **気象庁の警報がスコアに乗らない**(前提②)。暴風/波浪警報 + モデル凪 = 警報チップの横で「✅出港OK」。
3. **暴風がマスクされる**(前提④)。波が凪なら風20m/sでもGO。風の独立ゲートが無い。
4. **警報取得失敗が"平穏"に化ける**(前提③)。Web版はstateを出さない。
5. **推測座標のポイントで誤ったポイント別GO**(前提⑥)。

「データ取得失敗・凍結・遅延のときに危険が伝わらん経路」は**まさに前提①③⑧が該当**。失敗が「良好」に見え、古さが「たった今」に見え、警報欠落が「平穏」に見える——**3つの沈黙がすべてGO方向**に効く。これが段階1の最大の弱点。

### Q2(重点)安全設計と責任／有料化での変質
**あるべきリスク設計(最小):**
- **fail-to-caution へ反転:** 必須データ(波・風)が欠損したらスコアを出さず「⚠️ データ取得失敗——現地判断してください」を表示。GO寄りデフォルトを廃止(前提①)。
- **警報をスコアに直結:** 気象庁が暴風/波浪(特別)警報を出している間はスコア/バナーに天井(例:最大3・「要確認」以上にしない)。少なくとも**警報時にバナーをGOにしない**(前提②)。
- **免責の一文を常時表示:** 「本アプリは気象・海況の参考情報です。出港・ダイビングの最終判断はガイド本人の責任で行ってください。安全を保証するものではありません。」——今ゼロ(前提⑩)。フッターの「安全業務をサポート」という自称は、免責と併記しない限り危険。
- **失敗を正直に見せる:** 取得失敗・stale・データ時刻をWebでも明示(メール実装を流用)。

**「事故ったら誰の責任か」:** 現状、開発者(たかしさん)は**免責を一切置かず「安全ツール」を自称**している。無償・友人間でも、日本の民事では「情報提供者の注意義務」が問われ得る文脈を自ら作っている。**段階3の有料化で責任は質的に変わる:** 対価が発生すれば「商品として安全に資する情報を提供する」契約的期待と、消費者保護・PL的な視線が乗る。**免責文が無いまま課金する**——これが本アプリで最も危険な事業判断。有料化の前に、利用規約・免責・「参考情報」の明示は交渉の余地なく必須。

### Q3(簡潔)段階2ゲート
お客様に見せる条件の第一は**UIの美しさではなく免責と失敗時挙動**。根拠不明の数字(前提⑤)を素人のお客様が見れば「アプリが安全と言った」に化けやすい。ガイド向け=判断材料(生値+不確実性)、お客様向け=安心と期待感だが、**お客様向けほど断定を弱め、免責を強く**。順序は「安全側の作り直し → 見た目磨き」。今は逆。

### Q4(簡潔)段階3現実性
根本から変わるべきは:(a)スコア基準の外部妥当性(海事/事故データ・専門家レビュー)、(b)fail-safe思想、(c)ポイントDBの実座標検証と網羅、(d)警報連動、(e)利用規約/免責/課金基盤/サポート。**今から仕込むべき**は免責・失敗時挙動・警報連動(安いし全段階で効く)。**今やらんでええ**のは課金・多地域展開・凝ったUI(基盤が危険なまま拡げると事故が全国化する)。

### Q5(簡潔)残件の優先順位(★は優くんの希望度)
安全価値で並べ替え、希望度との食い違いを明示:
- **最優先(希望リストに無いが最重要):** ①免責文 ②欠損時fail-to-caution ③警報スコア連動。★は付いていないが**段階1の未達を埋める本丸**。
- ⑤ガイド間共有★4:安全価値中・実現コスト大(DB/認証)。段階2まで保留。
- ①透明度予報★5:希望度最高だが、**透明度は現状のどのAPIでも実測困難で、出せば前提⑤の"それっぽい嘘"が増える**。安全価値と実現性が★5に見合わない典型。作るなら「推定・参考」と明記必須。**希望度と実現性の最大の食い違い。**
- ③深度別水温★3・⑦オフライン★3:あれば便利、安全の本丸ではない。⑦は電波なし船上で価値あり(現場UXへ委譲)。
- ⑥お客様一発共有★2:免責が無いまま実装するとお客様に生スコアが拡散し責任面が悪化。免責整備後に。

### Q6(簡潔)開発体制の持続性
単一メンテナ・テスト不在・無償・非専門家。**個人プロジェクトへの過剰品質要求は戒める**——CI/CDフルテストは要らん。だが**安全に関わる1点(欠損時の挙動)だけはテストで固める**価値がある(calcScoreの境界・欠損入力のスナップショットテスト数本)。cron単一依存のデッドマンズスイッチも安い保険。バス係数1(たかさん倒れたら終わり)は無償なら許容だが、有料化するなら別問題。

## このアプリで最も危険な一点

**全API(または海況API)が落ちた瞬間、アプリは「スコア8 / ✅ 出港OK」を表示する。しかもエラー表示は出ない。**

検証(手計算・全データnull):
- `app.js:52-55` のデフォルト → 波1.0m(8点)・風10km/h(10点)・天気快晴(10点)・うねり8s(7点)
- raw = 8×0.4 + 10×0.35 + 10×0.15 + 7×0.1 = **8.9**、excess = 0.9、adjusted = 8.9 − 0.9×0.5 = **8.45 → 8点**
- `ui.js:44-45` → 8 ≥ 7 → バナー「✅ 出港OK」、`score-text`「✅ 良好なコンディション」、リングは緑
- `main()` の `catch`(`app.js:123-127`)は `Promise.allSettled`(`api.js:155`)が例外を握り潰すため**発火せず**、「データ取得に失敗しました」は表示されない
- 3地点cardは "--"、警報チップも空(=平穏に見える)。**だが最も大きく目立つリングとバナーだけが自信満々にGOと言う**

つまり **「データが1つも無い日」と「絶好の日」が、画面上ほぼ同じ緑のGO表示になる。** 台風接近で気象APIが不安定な日ほど、この誤GOが出やすい——最も危険なときに最も危険な嘘をつく構造。免責も無いので、これを信じて出港して事故れば、責任の所在は「安全ツールを自称した開発者」に向かう。**ここを直さずに段階2・3へ進むのは論外。**

## 切るべきもの / 足すべきもの

### 切るべき
- **GO寄りの欠損デフォルト**(`app.js:52-55`・`daily-brief.js:320-326`)。波・風が欠損したらスコアを出さない。
- **フッターの「最終更新=描画時刻」**(`ui.js:474-477`)。データ時刻に置換するか、いっそ消す(嘘より無)。
- **「安全業務をサポート」の自称を単独で置くこと**(`index.html:335`)。免責と必ずセットに。
- **透明度予報★5を今の技術で"予報"として出すこと**。実測できないものを断定調で出すのは前提⑤の悪化。

### 足すべき(安い順・全段階で効く)
1. **免責の一文**を今日ページ最下部と週間セクションに常時表示(コスト最小・効果最大)。
2. **欠損時 fail-to-caution:** 波 or 風が null ならスコア非表示+「⚠️ データ取得失敗——現地で判断を」。
3. **警報→スコア天井:** 気象庁の警報発表中はバナーをGOにしない/スコア上限を掛ける。
4. **Web版にもstale/失敗の明示**(メールの `buildWarningsSection` の分岐を流用)。
5. **風の独立ゲート**(例:風18m/s超は波と無関係にスコア上限)。
6. **cronデッドマンズスイッチ**(24時間メールが無ければ自分に警告)。
7. **calcScoreの境界・欠損入力テスト**を数本だけ(過剰品質でなく安全最小限)。

以上。数字がきれいなアプリほど、間違ったときに疑われん。まず「間違ったときに危険側へ倒れる」よう作り直してから、見た目とお客様と課金や。順序を守れ。
