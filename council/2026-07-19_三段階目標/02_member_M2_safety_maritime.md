## 担当レンズの深掘り — ⚓ 安全・海事リスク

海の現場の鉄則は一つ。**「情報が無い＝危険とみなす」**（fail-safe）。天気が読めない、無線が入らない、データが来ない——このとき海のプロは出港を見送る。本アプリの安全性は、この一点をコードがどう扱っているかで決まる。結論から言うと、**逆に実装されている**。

### 危険度A（誤GO・最重要）— データ全滅が「緑の出港OK」になる

`js/app.js` の `renderAll`（52-57行）:

```js
const currentWave  = kerama?.hourly.wave_height?.[hIdx]       ?? 1.0;
const currentWind  = (weather?.current?.wind_speed_10m ?? 10) / 3.6;
const currentCode  = weather?.current?.weathercode            ?? 0;
const currentSwell = kerama?.hourly.swell_wave_period?.[hIdx] ?? 8;
```

欠損時のフォールバック値が**すべて「良好側」**に倒れている:
- 波高欠損 → **1.0m**（波スコア8＝穏やか）
- 風速欠損 → **10km/h**（風スコア10＝ほぼ無風）
- 天気欠損 → **コード0**（快晴・スコア10）
- うねり欠損 → **周期8秒**（スコア7）

`fetchAll`（`js/api.js:154-163`）は `Promise.allSettled` を使い、**個々の失敗を握り潰して null を返す**。そして `main().catch`（`app.js:123-127`）はエラー時に「データ取得に失敗しました」と表示するはずだが、`fetchAll` は allSettled ゆえ**決して reject しない**。つまりこの catch は**永久に発火しない**。

結果、Open-Meteo が全滅（サーバ障害・オフライン・通信断）しても、アプリは例外を出さず、上の捏造値でスコアを計算する:

```
raw = 8×0.40 + 10×0.35 + 10×0.15 + 7×0.10 = 8.9
波高ペナルティ後 = 8.9 − 0.9×0.5 = 8.45 → 四捨五入で 8
```

**スコア 8 → `scoreLabel`（score.js:58）で「✅ 良好なコンディション」→ バナー（ui.js:44-46）で緑の「✅ 出港OK」**。

**通信が一本も繋がっていない状態で、アプリは自信満々に「出港OK 8/10」を表示する。** これが本アプリで最も危険な一点。船上は電波が切れる場所であり（優くんリスト⑦オフラインモードの本質はここ）、まさに現場で最も起きやすい障害が、最も危険な誤表示を生む。

補足の悪質さ: 3地点カードは `kerama ? ... : '--'`（`ui.js:142`）とガードしているので、**カードは「波 --」なのにヘッドラインは「出港OK 8/10」**という矛盾表示になる。ユーザーは「詳細は取れてないけど総合はOKなんだな」と誤読しかねない。

### 危険度B — 「凍結データ」が凍結と分からない

`js/score.js` の `findCurrentHourIndex`（115-119行）:

```js
const idx = times.findIndex(t => t.startsWith(nowStr));
return idx >= 0 ? idx : 0;   // 現在時刻が無ければ黙って先頭を使う
```

API が古いデータセット（現在時刻を含まない）を返した場合、**黙って配列先頭（＝別の日・別の時刻の値）でスコアを出す**。鮮度チェックが無い。

さらに `ui.js` の2つの時刻表示が食い違っている:
- `renderDataInfo`（482-490）は `weather.current.time`（API側の時刻）を表示するが、**それが「今」から何時間ずれているかの照合は無い**。
- `renderFooter`（473-478）は `new Date()`（＝**描画した瞬間の壁時計**）を「最終更新」として表示する。

つまり「最終更新 14:30」は**データの新しさではなく、画面を開いた時刻**を指す。データが3時間前のものでも「最終更新 14:30」と出る。これはユーザーの信頼を裏切る典型的なミスリード。bosai JSON が6週間凍結した実績（handover記載）を踏まえれば、Open-Meteo も同様に凍結し得る。そのとき `dataSignature`（`app.js:42-47`）は同じタイムスタンプを返し続けるので**更新ボタンも出ず**、ユーザーには「変化なし＝海況安定」に見える。**凍結は不可視**。

### 危険度C — 気象庁の公式「警報」がGO/NO-GO判定に一切効いていない

出港バナー（`ui.js:42-56`）は**スコアのみ**で決まる。警報（`renderWarningChips`・`renderDivePoints`）は**別枠のチップ**として並ぶだけで、バナーやスコアには一切反映されない。

`calcScore`（`score.js:32-53`）の入力は波高・風速・天気コード・うねり周期のみで、**警報コードは引数にすら入っていない**。

帰結: **「🌊 波浪警報」チップと「✅ 出港OK」バナーが同一画面に共存し得る**。波浪警報は気象庁が発表する公式・権威ある信号、スコアは Open-Meteo 由来の素人加重平均。両者は別ソース・別しきい値なので普通に食い違う。海事安全の観点では**これは順序が逆**。官の警報こそがスコアを上書き（ハードキャップ）すべきで、素人の点数が官の警報を黙殺してよいはずがない。

### 危険度D — 最も信頼する警報ルートに、最も防御が無い

`js/warnings.js` の `parseWarnings`（134行）:

```js
if (json.via !== 'xml') {          // XML経由は鮮度ガードを免除
  if (!Number.isFinite(reportTime) || now - reportTime > STALE_MS) {
    return { items: [], reportDatetime, stale: true };
  }
}
```

防災情報XML（第一候補・信頼ルート）は48時間鮮度ガードを**丸ごと免除**され、bosai JSON（予備・停止歴あり）だけがガードされる。

免除の理屈は「気象庁仕様＝新発表が無い＝前回発表が有効」であり、警報のセマンティクスとしては**正しい**。だが安全設計としての穴はここ: **JMAのXMLフィード自体が凍結した場合**（bosai がまさにそうなった）、「平穏で新発表が無い」状態と「配信が死んで新発表が届かない」状態が**コードから区別できない**。

危険な向きは「アンダー警報」: フィード凍結中に新しい波浪警報が発表されても、アプリは凍結前の「警報なし」を表示し続け、**ガードが免除されているため誰も鮮度を疑わない**。予備ルート（bosai）には残した48hガードという安全網を、より重要な第一ルートから**外してしまっている**。この非対称は見直すべき。

### 危険度E — 「点」の精度を偽装している

`js/config.js:20-32` の `DIVE_POINTS`。座標は開発者の推定で、**優くん確認済みはトライアングルのみ**（`config.js:22` の TODO コメント、handover 72行でも明記）。Marine API の格子は約5km。リーフや水路際のポイントでは、5km格子の1セルが**外洋と陰の水面をまたぐ**ことがあり、実際のダイブ地点を代表しない波高が返る。

にもかかわらず UI は各ポイントに**色付きの確定スコアチップ**（`ui.js:246` の `score-chip`）を表示し、あたかも地点固有の精度があるかのように見せる。**データが持たない精度を、見た目が主張している**。8ポイントのうち黒島北・シュガーヒル・クエフ北の3点は座標そのものが未確認（handover 72行）。

### 危険度F — 全フォールバックが「楽観」に倒れる構造的欠陥

危険度A・Bで見た通り、`app.js`・`ui.js`・`daily-brief.js` を通じて欠損時のデフォルトが**例外なく良好側**（波1.0m / 風10km/h / 天気快晴 / うねり8s）。安全システムの原則は「不明な入力は保守側（＝出港見送り側）に倒すか、判定自体を保留する」。本アプリには**保守側に倒すフォールバックが一つも無い**。これは個別のバグではなく**設計思想の欠陥**。

### 明るい点（安全設計として評価できるところ）

公平に記す。以下は海事安全の観点から**良い判断**:
- **波高の絶対キャップ**（`score.js:49-50`）: 波2.5m超は係数に関係なく最大2点。出港の一番のボトルネックを妥協せず固定したのは正しい。
- **メールの fail-safe**（`daily-brief.js:309-312`）: 天気・慶良間の必須データが取れなければ**メールを送らない**。捏造値でメールを送るより誠実。ただしこれは Web の fail-optimistic と**方針が真逆**（後述の矛盾）。
- **メール警報の正直な失敗表示**（`daily-brief.js:82-86`）: 鮮度ガード発動時に「更新停止中のため表示できません＋公式リンク」を明示。Web 版（`ui.js:176`「何も出さない」）より遥かに良い。この誠実さを Web にも移植すべき。
- **鮮度ガードという発想自体**（`warnings.js:120-122`）: bosai 凍結を経験して「古い警報で誤誘導しない」という安全網を設けた着眼は正しい。問題は第一ルートで免除したこと（危険度D）。

## 6問への回答

### Q1 — 段階1の達成度・誤GOの残存経路【重点】

**達成度: 部分的。日々の「気づき」の道具としては機能するが、「判断」の道具としては安全の向きが逆。**

誤GO（危険なのに出港OKに見える）の残存経路を、実コード根拠つきで列挙:

1. **【最重要】通信断・API全滅 → 「✅出港OK 8/10」**（`app.js:52-57` + `api.js:154-163` の allSettled + 発火しない `app.js:123` の catch）。船上の電波断という最頻の障害が、最悪の誤表示を生む。
2. **凍結データ → 別時刻の値でスコア算出**（`score.js:118` の `return ... : 0` 先頭フォールバック）。凍結は更新ボタンも出ず不可視（`app.js:42-47`）。
3. **「最終更新」が壁時計**（`ui.js:477`）で、古いデータを新鮮に見せる。
4. **波浪警報中でも出港OKバナー**が出る（警報がスコア・バナーに非連動、`score.js:32` の引数に警報なし）。
5. **XMLフィード凍結時のアンダー警報**（`warnings.js:134` のガード免除）。
6. **未確認座標×5km格子の点別スコア**が地点精度を偽装（`config.js:22`・`ui.js:246`）。

**データ取得失敗が危険として伝わらない経路**（設問後半への直接回答）: 上記1・2・3が該当。**Web 版には「データが取れていない」ことをユーザーに赤く伝える経路が存在しない**。`renderWarningChips`（`ui.js:176`）に至っては取得失敗時に「何も出さない（画面クリーン）」——**沈黙を『警報なし』と見分ける術がユーザーに無い**。これは危険な絶対的欠落（ambiguous absence）。

### Q2 — 安全設計と責任【重点】

**個人開発・無償アプリが安全判断の材料になることのリスク設計:**

まず思想を一本通す——**このアプリは「判断するもの（decision-maker）」ではなく「気づきを与えるもの（awareness aid）」に徹すべき**。現状は「✅ 出港OK」「🚫 出港困難」（`ui.js:44-52`）という**断定的な指示語**を使っており、これが責任を引き受けてしまう言葉になっている。海のプロの判断（潮・地形・客のレベル・その日の体調）を、4変数の加重平均が置き換えられるはずがない。

具体的にあるべき設計:

1. **免責の明示（今すぐ・コスト最小）**: 現状コード上に免責は事実上ゼロ。トップと各判定の近くに「**本アプリは参考情報です。最終的な出港判断は必ずガイド本人が現地の海況・気象庁の正式発表で行ってください**」を常時表示。免責は法的お守りである以上に、**アプリの立ち位置を「判断者」から「参考」へ降ろす宣言**として機能する。
2. **断定語を助言語へ**: 「✅ 出港OK」→「☀️ 条件は良好（要現地確認）」等。指示ではなく状態記述に。
3. **失敗時こそ雄弁に（fail-loud）**: データが取れないときは**灰色・?・警告**を出し、絶対に緑を出さない。捏造フォールバックを廃し、「判定不能」を第一級の表示状態にする。メール側（`daily-brief.js:82-86`）が既にやっている誠実さを Web に移植。
4. **官の警報を最上位に**: 気象庁の警報が出ていれば、スコアが何点でもバナーを NO-GO/要確認にハードキャップ。素人の点数が官の警報を上書きできない構造にする。
5. **優くん側の運用ルール**（アプリ外の約束事として明文化すべき）:
   - アプリの「出港OK」だけで出港を決めない。**必ず気象庁の警報・注意報の現物**（`index.html` から公式リンク）と、窓の外の実海況で最終確認。
   - 画面が「--」だらけ・スコアが妙に高い朝は**データ障害を疑う**。
   - お客様には「参考ツール」と説明し、判断はガイドがすると明言。

**段階3の有料化で責任がどう変質するか:**

無償の善意ツールは「使う側の自己責任」で概ね免責され得る。しかし**対価を取った瞬間、対価に見合う品質・安全の合理的期待**が生じ、製造物責任・消費者保護・利用規約の履行義務が発生する。「素人のバイブコーディング・テスト無し・単一メンテナ」（handover の体制）で全国の見知らぬガイドの出港判断を有料で支えることは、**開発者個人（本業FP&A・たかしさん一人）が負いきれない賠償リスク**を意味する。全国の誰かがこのアプリの「出港OK」を信じて事故れば、無償なら「参考情報でした」で済む余地があるが、有料なら「金を取って安全情報を売っておいて」の世界になる。

**したがって段階3は、(a) 中核ロジックの fail-safe 化、(b) 実効性ある利用規約・免責同意フロー、(c) データソースのSLA/冗長化、(d) 賠償を切り離す事業体（法人化・保険）——この4点が揃うまで着手すべきでない。** 今の延長線上で課金するのは、安全・法務の両面で無防備。

### Q3 — 段階2へのゲート（簡潔・担当外）

安全レンズからの条件のみ: **お客様の目に「出港OK/困難」の断定を見せる前に、免責と『参考情報』化を必ず済ませること**。客が画面を見て「アプリがOKって言ってた」とガイド判断に反論する導線を作ってはいけない。客向けは「今日の海の雰囲気・水温・見どころ」等の**期待感情報**に限定し、GO/NO-GOの判断表示はガイド向け画面に隔離するのが安全。UX詳細は M3 に委ねる。

### Q4 — 段階3の現実性（簡潔・担当外）

安全レンズからの一点のみ: 全国展開は**「慶良間という土地勘のある海」だから許された甘さ**（推定座標・素人しきい値）が通用しなくなる。知らない海のポイントDBを素人推定で埋めれば、危険度Eが全国規模で再生産される。全国版は**地点データの出所と精度保証**が製品の生命線。詳細は M5 に委ねる。

### Q5 — 残件の優先順位（簡潔・担当外）

安全価値での並べ替えのみ提示（★は優くん希望度）:
- **⑦ オフラインモード（★3）を安全観点で最優先級に格上げ**。理由: 危険度A（オフライン→捏造GO）の温床。「オフラインで綺麗に諦める（＝緑を出さない）」だけでも安全上の価値が突出。フル機能のオフライン化は不要、**『通信が無いと分かったら判定を止める』の実装で足りる**。
- ⑤ ガイド間共有（★4）・⑥ お客様共有（★2）は、上記の免責・fail-safe が済むまで**安全上むしろ危険**（誤GOを拡散する導線）。順序を後ろに。
- ①透明度予報（★5）・③深度別水温（★3）は安全リスクが低く、体感価値が高いので独立に進めてよい。
- 詳細な費用対効果は M5 に委ねる。

### Q6 — 開発体制の持続性（簡潔・担当外）

安全レンズからの一点: **テスト不在（handover）で最も怖いのは、スコアの回帰バグが静かに誤GOを生むこと**。過剰品質は不要だが、**`calcScore` の一点だけは値のスナップショットテスト**（代表10ケースの入出力固定）を入れる価値がある。安全の中核関数だけは「知らぬ間に挙動が変わる」を防ぐべき。詳細は M4 に委ねる。

## このアプリで最も危険な一点

**通信断・API障害・オフライン時に、アプリが例外を出さず、捏造したフォールバック値（波1.0m・風10km/h・快晴・うねり8s）で「✅ 出港OK 8/10」の緑バナーを表示すること。**

根拠: `app.js:52-57`（楽観フォールバック）＋ `api.js:154-163`（allSettled が失敗を握り潰す）＋ `app.js:123-127`（allSettled ゆえ発火しない catch）＋ `ui.js:44-46`（score≧7で緑「出港OK」）。

### 切るべきもの

- **楽観フォールバック（波1.0 / 風10 / 天気0 / うねり8）**（`app.js:52-57`, `ui.js:199-200`, `daily-brief.js:320,165`）。欠損は「判定不能」に倒す。数値を捏造してスコアを出す挙動そのものを廃止。

- **「最終更新＝壁時計」表示**（`ui.js:473-478`）。データ時刻と乖離しているなら「最終更新」の語を使わない。
- **取得失敗時に警報チップを『何も出さない』**（`ui.js:176`）。沈黙を「警報なし」と誤読させる。メール式（`daily-brief.js:82-86`）の「取得できませんでした＋公式リンク」に統一。

- **免責・『参考情報』宣言の常時表示**（現状コード上ゼロ）。トップと各判定近傍。段階2・3の前提条件。
