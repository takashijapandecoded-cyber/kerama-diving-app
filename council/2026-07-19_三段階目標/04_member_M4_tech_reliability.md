## 担当レンズ深掘り — 壊れ方の設計（fault design）と依存の寿命

### 1. 最大の技術的欠陥:「全データ失敗 → サイレント誤GO」（構造の非対称）

これが本答申の中心。**実測で確認した**。

- `api.js:fetchAll()` は `Promise.allSettled` を使うため **絶対に throw しない**。全API失敗でも `{ weather:null, kerama:null, ... }` を返して正常終了する。
- `app.js` の `main().catch(...)` が出す「データ取得に失敗しました。リロードしてください」は、**fetchAll が throw したときしか発火せん=データ障害では事実上デッドコード**。
- `app.js:52-57` の `renderAll` は null 入力に対しハードコードのデフォルト値を使う:
  - `currentWave = kerama?.hourly.wave_height?.[hIdx] ?? 1.0`（波1.0m）
  - `currentWind = (weather?.current?.wind_speed_10m ?? 10) / 3.6`（風10km/h）
  - `currentCode = ... ?? 0`（快晴）
  - `currentSwell = ... ?? 8`（うねり8s）
- **このデフォルト値の組み合わせを `calcScore` に通すと結果は「8点」**。`node` で実測:

  ```
  全データ取得失敗時のスコア = 8 / ラベル = ✅ 良好なコンディション
  バナー判定: score>=7 → ✅ 出港OK
  ```

つまり **Open-Meteo が落ちる・回線が死ぬ・APIが全滅した日、優くんの画面には大きく「✅ 出港OK スコア8」が出る**。3地点カードは「--」になるが、人間は真ん中の巨大なリングとバナーを見る。

決定的なのは、**メール版(`daily-brief.js:309`)は正しくガードしとる**こと:
```js
if (!weather || !kerama) { console.error('必須データ取得失敗…送信スキップ'); process.exit(1); }
```
同じロジックを共有しとるのに、**失敗時の扱いだけが web とメールで非対称**。これは単なるバグでなく「壊れ方の設計」がチャネル間で統一されとらん構造問題。しかも **フォールバックの思想が「好天のデフォルト値で埋める」= 本質的に危険な向き**（安全側=悪天候に倒すべき局面で、好天に倒しとる）。

**デリスク:** メール版のガードを web に横展開するだけ。「必須データ(`kerama`/`weather`)が null ならスコア領域を『⚠ データ取得不可・判断保留』にする」ゲートを `app.js` に1本足す。工数は小、効果は最大。**これが本アプリで最速のデリスク一手**。

### 2. 「取得失敗」と「平穏」が区別できん（警報のサイレント欠落）

- `ui.js:renderWarningChips` は `items` が空なら `box.innerHTML = ''`。コメントも「発表なし or 取得失敗 → 何も出さない」と自認しとる。
- 海況ページのポイント行も同じ。`warnings` が null なら警報ラベルが出ないだけ=警報ゼロと**同じ見た目**。
- つまり **XML と bosai の両ルートが落ちた日、警報が実際に出ていても、優くんの画面はクリーンなまま「平穏」に見える**。
- メール版は `stale` を「更新停止中のため表示できません」と正直に出す（`daily-brief.js:82`）が、**web にはこの正直さが無い**。ここでも web/メールの非対称。

### 3. 依存APIの寿命リスク（構造レベルの棚卸し）

| 依存 | 役割 | 寿命リスク | 安全影響 |
|---|---|---|---|
| **Open-Meteo**(marine+weather) | スコアの心臓 | **無料枠は非商用限定**（段階3で規約違反）。SLA無し・予告なくレート制限/仕様変更あり。8点マルチ座標は Open-Meteo 独自機能でロックイン | 致命的 |
| **気象庁 bosai JSON** | 警報（現・予備） | **5/28から6週間凍結の実績**。「無告知で止まるソース」に依存した前例そのもの。予備降格は正しい対応 | 中 |
| **気象庁 防災情報XML**（現・第一候補） | 警報（正規配信） | **正規表現でスクレイピング**（`_VPWW54_471000`・`気象警報・注意報（市町村等）`・`<Kind><Code>`）。気象庁がH27スキーマを改訂したら正規表現が全滑り→**警報が静かに消える**。しかも失敗はサイレント（上記2） | 高 |
| **NASA EPIC / DEMO_KEY** | 背景画像 | `config.js:46` に `DEMO_KEY` 直書き=共有キー・日40回。他人の消費で枯れる | 無（背景が出ないだけ） |
| **Chart.js CDN(jsdelivr)** | 潮汐グラフ | SRI+defer済みで供給網対策は妥当。jsdelivr障害でグラフ欠落のみ | 低 |
| **nodemailer + Gmailアプリパスワード** | 朝メール | 単一経路。Gmailのアプリパスワード廃止・送信制限(1日500通)・アカウントロックで**黙って止まる**。多人数化で即詰む | 中（朝メール欠測） |
| **cron-job.org**(無料)→workflow_dispatch | 朝メールのトリガ | **無料外部cronが単一障害点**。二重起動回避で GitHub schedule を削除した結果、ここが落ちると朝メールが飛ばず、**「飛ばなかったこと」を検知する仕組みが無い**（サイレント欠測） | 中 |

**構造的教訓:** bosai凍結が示したのは「気象庁でさえ無告知で止まる」という事実。**単一ソース依存は全部同じ脆さを抱えとる**。今の設計は変換層で乗り換え耐性の芽はあるが、**失敗を検知する層（監視）が完全に欠けとる**ので、次の凍結が起きても「たかしさんが偶然気づくまで壊れたまま」。

### 4. 鮮度ガードの非対称という副作用

`warnings.js:134` — XML経由は鮮度ガード(48h)を**免除**する。「新発表なし＝前回発表が有効」という気象庁仕様に基づく判断で、**それ自体は正しい**。ただし裏を返すと:
- XMLルートには**古いデータを検知する上限が無い**。`memoSet` で覚えたURLを使い回す設計上、「気象庁側で消えずに残った古い警報XML」を掴み続けても48h検知が効かない。
- bosai だけが48hガードの保護下。第一候補のXMLは「気象庁が新発表を出し続ける」ことに全面依存。
- 実害は低いが、「ガードの適用ルールが配信方式で分岐する」複雑さは、将来メンテナが変わったときの誤解の温床。

### 5. テスト不在 — ただし戒めつき

- **テストは1本も無い**（root に package.json すら無く、`*.test.js`/eslint 皆無を確認）。
- しかし `score.js`(calcScore) と `warnings.js`(xmlToWarningJson/parseWarnings/鮮度ガード) は **副作用のない純粋関数**で、**テスタビリティは極めて高い**。テストを書かん理由が無い種類のコード。
- この2つは「静かに壊れる」代表格。スコア係数を触る・XMLパーサを触るたび、今は **`curl 本番 | diff ローカル`(handover)の人力目視**で回帰検証しとる。パーサの取りこぼしは目視では気づきにくい。
- **戒め:** 全面的なCI/E2E/型システムは段階1には過剰。要らん。**入れるべきは score.js と warnings.js のパーサのユニットテストだけ**。数十行・数時間・依存ゼロで、費用対効果が全依存の中で最大。「網羅的テスト基盤」を目指すのは段階3が見えてから。

### 6. 単一メンテナ=バス係数1

- 開発者はたかしさん一人・本業別・無償。**バス係数1**。
- bosai→XML乗り換えを実際にやり切った実績はある。だがそれは「たかしさんが凍結に気づき、手を動かせたから」成立した。**旅行・多忙・体調・離脱で、壊れたまま誰も気づかない期間が必ず生じる**。
- 段階1（優くん一人・無償・本人も海のプロ）ならこのリスクは**許容範囲**。優くんは自分の目と経験で最終判断できる。
- **問題は段階2**。お客様という「アプリを鵜呑みにしうる第三者」が入った瞬間、単一メンテナ+サイレント障害+監視ゼロの組み合わせが「重大リスク」に変質する。段階2のゲートは新機能でなく**この体制の底上げ**。

## 6問への回答

**Q1（段階1達成度・誤GO経路）— 担当・技術面:**
機能面は段階1に十分応えとる。だが**誤GOの技術的経路は残っとる**。(A) 全データ失敗→デフォルト値で8点「出港OK」（上記深掘り1・実測済み）。(B) 部分失敗の混在（divePointsだけ失敗しても総合スコアは平然と算出、どのデータが欠けとるか画面から読めん）。(C) 警報の取得失敗が「平穏」と区別つかず、警報を握りつぶす（深掘り2）。**「データが無い」ことが「良い」と誤変換される**のが構造上の穴。優くんが海のプロで最終判断できるうちは致命傷にならんが、技術的には未完成。

**Q2（安全設計と責任）— 簡潔:**
`index.html` 全文を確認したが**免責表示がどこにも無い**（「参考」の語は週間カレンダーの「お客様連絡の参考に」だけ）。安全判断材料になるアプリで免責不在は段階2前の必須整備。ただし**免責文言より先に「失敗時に嘘のGOを出さない」技術ゲート（Q1-A）のほうが優先**——免責は事故後の法的保険、ゲートは事故そのものの予防。有料化すると「対価をとった情報提供」になり、可用性・正確性の責任が跳ね上がる=技術的信頼性の底上げが有料化の前提条件になる。

**Q3（段階2ゲート・UX）— 技術面のみ簡潔:**
オフライン（残件⑦）は現状 sessionStorage キャッシュのみで **PWA/Service Worker 無し**。船上=電波なしでリロードすると全部消える。技術的には Service Worker + マニフェストで「最後に取れた値のオフライン表示」は低コスト実現可能。ただし**「これは何時時点の値か」を明示せんと、また別の誤GO経路を作る**（古いキャッシュを最新と誤認）。

**Q4（段階3の現実性）— 簡潔:**
根本から変わるべきは3点。① **Open-Meteo無料枠から離脱**（商用契約 or 気象庁/民間気象へ。規約・コスト・SLA）。② **ポイントDBの構造化**（現状 `config.js` に8点ハードコード・座標も大半が推定。全国はDB必須）。③ **多人数配信基盤**（nodemailer 単発では詰む・認証も要る）。今から仕込むべきは「データソース抽象化層」の意識的維持のみ（warnings で変換層を挟んだ実績=既に正しい方向）。課金・スケール対応は今やらんでええ。

**Q5（残件優先度）— 技術的実現性の観点:**
希望度と技術的実現性が**大きく食い違う**。①透明度予報★5 は **そもそも Open-Meteo に透明度データが存在しない**公算が高く、実現性は最低——★5だが「作れるか」を先に検証すべき。③深度別水温★3 も Open-Meteo は表層SST止まりで、別ソースが要る。逆に **⑦オフライン★3 は地味だが確実に作れて、誤GO予防(Q1)にも効く**ので技術的コスパは最上位。「欲しいものはデータが無く、確実に作れるものは地味」という構図を優くんに正直に伝えるべき。

**Q6（体制の持続性）— 担当:**
上記深掘り3〜6。要点は「今の身の丈構成（静的サイト+GitHub Actions+ロジック共有）は段階1には適切で、過剰堅牢化は要らん。足すべきは①失敗時ゲート ②パーサ/スコアの最小テスト ③朝メール欠測とAPI失敗の最小監視——の3点だけ」。単一メンテナは段階1では許容、段階2で重大リスクに変質する。

## このアプリで最も危険な一点

**全データ取得失敗時のサイレント誤GO。** メール版は `!weather || !kerama` でガードしとるのに、web版だけが「波1.0m・快晴・うねり8s」のハードコードデフォルトで **8点「✅ 出港OK」を表示する**（実測確認済み）。Open-Meteo障害・回線障害・海底ケーブル障害などでデータが全滅した日、優くんの画面に嘘のGOが出て、**しかも「静かに壊れる」ので誰も気づかない**。デリスクはメール版のガードをweb(`app.js`)に横展開する1点で足りる。

## 切るべきもの / 足すべきもの

**切る:**
- **過剰堅牢化の構想**（フルCI・E2E・型システム・スケール設計）は段階1では全部切る。身の丈を超える。
- **DEMO_KEY依存のNASA EPIC背景**は「あれば嬉しい」枠。段階2以降で信頼性を語るなら、共有キーで枯れる要素は自前キー化 or 撤去候補（安全機能ではないので優先度最下位）。

**足す（優先度順・段階1のうちにやる価値がある順）:**
1. **失敗時ゲート**（必須データnull→スコア領域を「取得不可・判断保留」に）＋**警報の「取得失敗」と「平穏」の区別**。← 最速デリスク。web/メールの非対称を解消。
2. **score.js / warnings.js パーサの最小ユニットテスト**（純粋関数・低コスト・回帰の早期検知）。
3. **朝メール欠測とAPI失敗の最小監視**（cron-job.org の dead-man 監視、GitHub Actions 失敗通知）。「静かに壊れる」を「気づける」に変える。
4. **免責表示**（段階2ゲート・法的保険。ただし1より後でよい）。
5. **データソース抽象化層の意識的維持**（段階3の乗り換え耐性。既に芽はある）。

段階1の完成度を上げる最短経路は、新機能でなく **「壊れ方を正直にする」1〜3**。ここを固めん限り、段階2で他人（お客様）の安全に踏み込むのは早い。
