## 1. 一致点(全員または過半数)

### 1-1. 【6/6・最重要】データ全滅で「✅ 出港OK 8点」が出る — サイレント誤GO

全議員が独立に同一の欠陥に到達した。M1・M2は手計算、M4はnode実測、M6は静的解析で、**同じ「8点」を算出**。

- `app.js:52-57` の欠損フォールバックが全項目「良好側」: 波`?? 1.0`m・風`?? 10`km/h・天気`?? 0`(快晴)・うねり`?? 8`s
- `api.js:154-163` の `Promise.allSettled` が失敗を握り潰すため、`app.js:123-127` の `main().catch` は**永久に発火しないデッドコード**
- 結果: 全API障害・回線断・圏外で raw=8.9 → 8点 →「✅ 出港OK」緑バナー。エラー表示なし
- M6はさらに `calcScore({})` 単体が **10点満点**を返すことを確認
- 対照的に**メール版は正しくガード済み**(`daily-brief.js:309` `!weather || !kerama` → 送信スキップ)。webとメールで「壊れ方の設計」が非対称(M4)

M2の言葉: 「海のプロの鉄則は『情報が無い=危険とみなす』。本アプリは**その完全な反対**を実装しとる。船上の電波断という最頻の障害が、最悪の誤表示を生む。」

### 1-2. 【5/5】免責が一文も無い。逆に「安全業務をサポート」と自称しとる

M1〜M5全員がgrepで確認: 免責・「参考情報」・「最終判断はガイド」の類は**コード上ゼロ**。唯一の「参考」は `index.html:183`「お客様連絡の参考に」(用途説明であって免責ではない)。一方 `index.html:335` は「優くんの安全なダイビングガイド業務をサポートするために作られました」と**責任を引き受ける方向のコピーだけがある**。段階2・3に進む前の交渉不可の前提条件で全員一致。

### 1-3. 【5/6】気象庁の警報がスコア・バナーに一切効かん

`calcScore`(`score.js:32`)の引数に警報が存在せず、**暴風警報+モデル凪=「✅出港OK」バナーと警報チップの同居**が起こる(M1②・M2危険度C・M4・M6重大5)。官の警報が素人の加重平均に黙殺される構造は本末転倒で、警報中はバナー・スコアに天井(ハードキャップ)を掛けるべきで一致。

### 1-4. 【4/6】警報の「取得失敗」と「平穏」が区別できん

`ui.js:174-178` は失敗でも平穏でも空欄。M6が最も精密: 戻り値を `ok / none / stale / error` の4状態に分けよ。XML部分解析失敗が空配列=正常として通る穴も指摘。メール版は「更新停止中のため表示できません」と正直に出す(`daily-brief.js:82-86`)のに**webに同等の分岐が無い**(M1③・M2・M4)。

### 1-5. 【6/6】段階の順序: 安全固め → 段階2は「翻訳と免責」→ 段階3は今やらない

- **段階2の壁は機能ではなく言葉**(M3): 現UIは100%ガイドの業務言語(「出港困難」「キャンセル推奨」)。お客様向けは判断語を排した「体験語」への**翻訳ビュー**+免責で越えられる
- **段階3は別事業**(M5・M4): 全国ポイントDBは「データ事業」、課金はバックエンド一式+法務+サポート体制。さらに**Open-Meteo無料枠は非商用限定=今の構成のまま有料化すると規約違反**(M4・M5が独立に指摘)。競合は海快晴(全国約8,000地点・月330〜480円)が既に存在
- 唯一のクサビは「ダイビング専用のGO/NO-GO判断を肩代わりして押し出す」レイヤーやが、**その判断レイヤーは未検証でコピー可能**(M5)

### 1-6. 【4/6】唯一積み上がる資産=「優くんの実判断 vs スコア」の突き合わせ記録。今は1バイトも無い

M5の中心論点で、M1(体感フィードバックが1人・主観・未記録)・M2(自作しきい値が事故データ未照合)・M3(実測透明度の記録)が同旨。**同じ一手(毎日の記録)が安全検証とmoat構築の両方を救う**のに、毎日捨てとる。

### 1-7. 【4/6】テストは最小限だけ — calcScore と warnings パーサ

全面CI/E2Eは過剰で不要、ただし安全中核の純粋関数2つ(`score.js`・`warnings.js`)だけはユニットテストを書く価値あり、で一致(M1・M2・M4・M6)。M6が具体的なテスト3本を設計済み(答申06参照)。

### 1-8. 【3/6】精度の偽装 — 推定座標×5km格子×確定スコアチップ

8点中5点が推定座標(優くん確認済みはトライアングルのみ、`config.js:21-22` TODO)なのに、色付き確定チップと小数1位の潮流表示が**データが持たん精度を見た目で主張**しとる(M1⑥・M2危険度E・M5)。

## 2. 対立点と議長裁定

### ① 断定語「出港OK」を捨てるか残すか(M2 vs M5)

- M2: 「✅出港OK/⛔キャンセル推奨」は判断の権威をアプリに移す危険な指示語。「状態記述+要現地確認」へ降ろせ
- M5: GO/NO-GO判断の肩代わりこそ**このアプリ唯一の防御可能な価値**。データ表示に退化したら海快晴・Windyに存在意義で負ける

**裁定: 両立させる。** ガイド向け(段階1)は判断の鋭さを残す — ただし(a)気象庁警報がバナーを上書きするハードキャップ、(b)データ欠損時は判断自体を出さない、(c)「要現地確認」の常時併記、の3条件つき。お客様向け(段階2の翻訳ビュー)では判断語を**全面排除**。つまり「判断を名乗るのは、正直に判断できるときだけ」。

### ② ⑦オフラインモードの優先度(M2「安全最優先級」 vs M3「使用実態を確認してから」)

- M2: 圏外→捏造GOの温床。「通信が無いと分かったら判定を止める」だけでも突出した安全価値
- M3: 優くんが船上でアプリを開くのか、5時メールで足りとるのかが**未検証**。メール派なら作っても空振り

**裁定: ⑦を「安全の半分」と「便利の半分」に分割する。** 安全の半分(圏外・欠損で緑を出さない)は⑦を待たず**アクション1のフェイルセーフ化に内包**して今やる。便利の半分(最終取得値+取得時刻の表示、SW軽量版)は**優くんに使用実態を聞いてから**GO/NO-GO。M3の指摘どおり、テキストメールは受信済みなら圏外でも読めるので、実は現状でも「オフラインで生き残る」媒体は存在しとる。

### ③ 免責と技術ゲートのどちらが先か(M4 vs M1/M2/M3)

- M4: ゲートが先(免責は事故後の保険、ゲートは事故そのものの予防)
- M1/M2/M3: 免責が先(一文でゼロコスト・効果最大)

**裁定: 偽の二者択一。同じパッチで同時に出す。** 免責はコード不要の一文、ゲートは数行。分ける理由が無い。

### ④ XML鮮度ガード免除の是非(前回セッションの決定 vs M2/M4/M6)

前回(7/19未明)の決定は「XML経由は鮮度ガード免除(気象庁仕様: 新発表なし=前回発表が有効)」。今回、3議員が独立にこの決定の穴を指摘:

- M2危険度D: フィード自体が凍結したら「平穏」と「配信死」が区別できず、凍結中の新警報を見逃す(アンダー警報)。bosaiがまさにそう死んだ
- M6重大3(Codex固有): `warnings.js:97-117` は**sessionStorageの記憶URLを長期フィードより先に無条件採用**するため、記憶URLが古いと火曜発表の新警報を無視して月曜の状態を出し続ける具体的シナリオがある
- M4: 「ガードの適用ルールが配信方式で分岐する」複雑さ自体が誤解の温床

**裁定: 前回決定を部分的に見直す。** セマンティクス(新発表なし=有効)は維持しつつ、(a)記憶URLの優先順位を「短期→長期→記憶」に変更(記憶は本文キャッシュにのみ使う)、(b)フィード自体の更新時刻で生存確認を足す。**前回セッションの決定の修正になるため、たかしさんの裁可対象。**

### ⑤ 段階1の点数(5点か7.5点か)

対立ではなくレンズの差と裁定する。**「晴れの日は7.5、雨の日は3」が同一アプリの実像。** 議長としての段階1総合は**条件付き5点**: アクション1・2(フェイルセーフ+免責)が入った時点で7点台の実力が「本物になる」。

## 3. 却下された前提(このアプリから剥がす札・8枚)

1. **「データが取れんときは無難なデフォルトで埋めればええ」** → 安全アプリでは最悪の設計。欠損は「判定不能」と正直に言う(6/6)
2. **「エラーはcatchが表示してくれる」** → allSettledのせいで永久に発火せんデッドコード(M2・M4・M6)
3. **「警報チップを出しときゃ伝わる」** → スコア非連動では、最も権威ある危険信号が最も目立つ数字に負ける(5/6)
4. **「最終更新=描画時刻でええ」** → 3時間前のデータも「たった今」に見える鮮度偽装(`ui.js:474-477`、M1⑧・M2)
5. **「朝5時の値=今日の出港判断」** → 5時凪・10時波3mの日に良好判定が出る。営業時間帯(7〜16時)の最悪値で判断すべき(M6重大2・M1⑨)
6. **「①透明度予報★5が次の目玉」** → 予報ソースが存在せん。数値で出せば当てずっぽう→信頼崩壊。定性ラベル+実測記録に再定義(M3・M4・M5一致)
7. **「段階3は今のアプリの延長」** → 別事業(データ事業+課金+法務+体制)。しかもOpen-Meteo無料枠は非商用限定で現構成の有料化は規約違反(M4・M5)
8. **「スコアは係数をいじれば当たるようになる」** → 検証データがゼロのまま調整を重ねても当てずっぽうの精緻化。突き合わせ記録が先(M5)

## 4. 判定① 今すぐ着手すべきアクション3つ

### アクション1 — 安全パッチ「嘘をつかないアプリ化」(コード・半日規模)

1. **必須データゲート**: メール版のガード(`daily-brief.js:309`)をwebへ横展開。`kerama`/`weather` が null ならスコア・バナーを出さず「⚠ データ取得不可 — 気象庁と現地で判断を」を表示。GO寄りデフォルト(`?? 1.0` 等)を廃止
2. **警報ハードキャップ**: 波浪・暴風(特別)警報の発表中はスコアに上限を掛け、バナーを絶対にGOにしない(web/メール共通の判定関数で)
3. **警報の4状態表示**: `ok/none/stale/error` を区別し、取得失敗・staleをwebでも明示(メールの誠実さを移植)
4. **同梱の小修正**: `findCurrentHourIndex` の先頭フォールバック廃止(`score.js:115-119`)・フッター「最終更新=壁時計」をデータ時刻に修正(`ui.js:474-477`)

### アクション2 — 免責と文言(コード不要・即日)

- 常時免責文: 「本アプリは気象・海況の参考情報です。出港・ダイビングの最終判断は必ずガイド本人が気象庁の発表と現地の海況で行ってください。安全を保証するものではありません。」を今日ページ最下部+メール末尾へ
- 「安全業務をサポート」(`index.html:335`)は免責と必ず併記
- バナーに「要現地確認」を常時併記

### アクション3 — キャリブレーション記録の開始(フォーム1枚・資産)

優くんの毎日の実判断(出た/やめた・体感スコア・実測透明度)をGoogleフォーム等で拾い始める。**スコア検証(安全)と段階3唯一のmoat(戦略)を同じ一手で獲る。** M5: 「今それをやっとらんのが最大の機会損失」。
